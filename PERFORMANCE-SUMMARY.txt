5. SUBPROCESS EXPLOSION: 8-30+ Processes per Update 💥
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EXECUTION CHAIN: plan-db-safe.sh update-task 123 done "msg"

  bash (main)
  ├─ plan-db-safe.sh
  │  ├─ sqlite3 (circuit breaker check)          [1]
  │  ├─ sqlite3 (check rejection count)          [2]
  │  ├─ sqlite3 (update if blocked)              [3]
  │  ├─ sqlite3 (audit log)                      [4]
  │  └─ flock (atomic write)                     [5]
  │
  └─ plan-db.sh (via exec)
     ├─ 11 source operations (no exec)
     └─ cmd_update_task()
        ├─ sqlite3 (check task exists)           [6]
        ├─ sqlite3 (validate status transition)  [7]
        ├─ sqlite3 (UPDATE tasks status)         [8]
        ├─ sqlite3 (check wave completion)       [9-12]
        ├─ jq (parse JSON if provided)           [13]
        └─ possibly: git (worktree validation)   [14+]

SUBPROCESS COUNTS:
  Simple (plan-db list):        2-3 processes
  Normal (update-task):         8-15 processes   ⚠️ Heavy
  Bulk (add-task w/ preconditions): 30+ processes ⚠️ Very Heavy

PROBLEM: Many single-row SQLite operations.
  • plan-db-import.sh: 51 sqlite3 invocations for bulk insert
  • plan-db-update.sh: 40 sqlite3 invocations for single update
  → Each call: fork + parse + execute + exit

QUICK WIN:
  • Batch SQLite queries (single sqlite3 process, multiple statements)
  • Use transactions to reduce commits
  • Cache worktree lookups (currently 2-3 queries per op)
  • Reduce: 30 calls → 5-10 calls (80% fewer subprocesses)

6. DUPLICATE QUERY LOGIC: No Query Builder ⚠️
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QUERY DISTRIBUTION:
  plan-db-import.sh:   51 sqlite3 calls
  plan-db-update.sh:   40 sqlite3 calls
  plan-db-create.sh:   14 sqlite3 calls
  plan-db-display.sh:   5 sqlite3 calls (SELECT v_kanban)
  plan-db-read.sh:      3 sqlite3 calls
  ─────────────────────────────────
  Total across libs:  ~150+ inline SQL calls

PROBLEM 1: No centralized SQL builder
  • Each SELECT written manually
  • sql_escape() called inconsistently
  • Vulnerability: SQL injection if escape missed
  • Duplication: Same queries written 3+ times

PROBLEM 2: Single-call pattern (no batching)
  sqlite3 "$DB_FILE" "INSERT INTO tasks..."  # Fork 1
  sqlite3 "$DB_FILE" "INSERT INTO tasks..."  # Fork 2
  sqlite3 "$DB_FILE" "INSERT INTO tasks..."  # Fork 3
  sqlite3 "$DB_FILE" "SELECT COUNT(*) ..."   # Fork 4

  Better:
  sqlite3 "$DB_FILE" << EOF
  BEGIN TRANSACTION;
  INSERT INTO tasks...
  INSERT INTO tasks...
  INSERT INTO tasks...
  SELECT COUNT(*)...
  COMMIT;
  EOF  # 1 fork, 1 commit cycle

QUICK WIN:
  • Create plan-db-query.sh helper module
  • Implement sql_insert_batch(), sql_update_batch()
  • Use transactions by default
  • Reduce: 51 calls → 10-15 calls (70% fewer)
